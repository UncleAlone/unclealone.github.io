1 Inheritance

Using inheritance is to avoid repeating the same/common parts(attributes and methods) in similar classes

1.1 Base class and subclasses

base class is the class which handle all the common things and we only need to write them only once

Subclasses* is a kind of the */base class/ and it is used to store the different parts

Example:

  • Base class: Animals
    • Subclasses: Dogs, Cats, etc.

1.2 Abstract classes

Abstract classes are the explicit representation of an interface in a Python program

An abstract class (as with all superclasses) also enables the sharing of code through method inheritance

  • Some methods will be unimplemented and are to be implemented by subclasses
  • For an abstract method, raise NotImplementedError
  • If some behaviours are identical in subclasses, those method can be implemented in an abstract class

Example: 

class Animals:

    def eat(self) -> str:
	raise NotImplementedError

class Dogs(Animals):

    def eat(self) -> str:
	return "eating meat..."

class Rabbits(Animals):

    def eat(self) -> str:
	return "eating carrot..."

1.3 Class design with inheritance

  • The four cases of method inheritance
    1. Subclass inherits superclass methods
    2. Subclass overrides an abstract method (to implement it)
    3. Subclass overrides an implemented method (to replace it)
    4. Subclass overrides an implemented method (to extend it)

1.4 Write general code

  • Client code writted to use base classes will work with their subclasses
  • The client code can rely on the subclasses

1.5 Same codes, different types

"One code to rule them all"

Terminology: polymorphism ("taking multiple forms") 

class Zoo:
    """
    ...
    """
    animals: List[Animals]
    ...

    def eat_all(self) -> None:
	for animal in self.animals:
	    animal.eat()

1.6 Avoid Duplicate Documentation

  • Do not maintain documentation in two places, e.g. superclass and subclass (unless there's no differences)
  • For inferited methods, common public attributes, there is no need to document again in subclass.
  • Sometimes some differences may need to explained
  • Remember: docstring is part of the public API, which should contain how to use a method instead of how it is implemented internally

1.7 Is a vs. Has a

Inheritance and composition is not always appropriate to describe the logical relationship between the entities you want to model.

  • Inheritance: "is a" relationship
  • Composition: "has a" relationship
